External design mockups (HTML + JSX exports from Claude Design / Stitch / Figma AI) live in `.local/design-mockups/` (relative to repo root) — gitignored via `/.local/` in `.gitignore`. Adrian explicitly does not want them committed to the repo.

**Why:** The repo is the production code source of truth. Mockups are pre-implementation artifacts that get translated to ERB views during execution. Keeping them locally available speeds up the implementation pass but committing them would pollute the repo with intermediate state that decays as the actual implementation diverges.

**How to apply:**

- When Adrian exports new mockups, expect them at `.local/design-mockups/<BatchName>/<screen_name>/`. The current batch is `Stockerly-1.0/` (placed there 2026-05-16).
- When implementing a screen, open the mockup as reference (`Read` tool works with absolute paths inside `.local/`). Do not copy-paste it verbatim; translate the structure to ERB partials reusing existing components.
- **Lumen palette is the source of truth** ([[stockerly-working-method]] → `docs/design/brand.md` §1 and `docs/design/tokens.md` §1). Mockups may drift on the margin (extra tokens, slightly different shadows, opinionated naming). In conflict, the implementation follows Lumen and the mockup loses.
- After implementing a screen, do NOT delete its mockup from `.local/design-mockups/` — keep as historical reference for QA / retro / future iteration.

### Lumen-fidelity audit (done 2026-05-16)

Checked `Stockerly-1.0/privacy/colors_and_type.css` as representative sample. **Honors Lumen completely** — `--color-primary: #5B6CFF`, `--color-bg-canvas: #FAFAF7`, `--color-fg-default: #0F172A`, all semantic tokens, dark mode block, Plus Jakarta Sans + Inter + JetBrains Mono. ✓

**Extras the mockup adds (not in `tokens.md` yet — consider promoting to canonical Lumen at S08):**

- `--color-chip-bg: #F1EEE6` (slightly warmer than `bg.muted`, used for pills/chips)
- `--color-primary-tint`, `--color-positive-tint`, `--color-negative-tint`, `--color-warning-tint` — translucent rgba overlays for hover/chip backgrounds (`rgba(91, 108, 255, 0.08)` etc.)
- `--color-primary-ink: #FFFFFF` (foreground on primary fill — explicit alias of `fg-inverse`)
- `--radius-card: 14px` (custom card radius, between `radius-md: 9px` and `radius-lg: 12px`)
- Type role `overline`: 11px, uppercase, letter-spacing 0.12em — for small caps section labels
- Motion tokens: `--motion-fast: 120ms`, `--motion-base: 200ms`, `--motion-slow: 320ms`, with cubic-bezier easings
- Z-index scale: `--z-nav: 40`, `--z-overlay: 80`, `--z-modal: 90`, `--z-toast: 100`

**Decision for S07:** when implementing, prefer existing Lumen tokens (`bg.muted`, `radius.lg`) where the mockup's extras don't add value. For mockup extras that are genuinely useful but not yet canonical:
- **Scalar values** (motion durations in ms, z-index integers): inline directly in ERB via `style="transition-duration: 200ms"` or via Tailwind arbitrary values like `z-[80]`. These are cheap and don't pollute the palette.
- **Colors / rgba overlays**: never inline as raw hex/rgba. Use Tailwind arbitrary values like `bg-[rgba(91,108,255,0.08)]` so the consumer code stays grep-able and migration to a canonical token (when promoted) is a single search-replace.

Token expansion to canonical Lumen is an S08+ decision via ADR.

## Workflow when implementing a screen with mockup

1. Open the mockup's `.html` in browser (or read the `.jsx` if no display available) to understand the visual hierarchy.
2. Read the issue's design-prompt comment to recall the constraints ([[pr-review-workflow]] references the prompts in #73, #74, #77).
3. Translate structure to ERB partials, reusing existing components from `docs/design/components.md` and view helpers.
4. Apply Lumen tokens through Tailwind utility classes (`bg-bg-surface`, `text-fg-default`, etc.). **Do not** copy raw hex values from the mockup.
5. Render both light and dark mode — manually verify in browser.
6. If the mockup conflicts with brand voice (`docs/design/brand.md` §2), the brand wins — descriptive copy is non-negotiable.
7. Screenshot the implementation and save to `docs/screenshots/` (per `docs/sprints/_template/qa.md`) — those screenshots ARE committed to the repo.
